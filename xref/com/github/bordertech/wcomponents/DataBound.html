3   /**
4    * <p>
5    * As a general concept all WComponents can be looked upon as being editors for data structures that are entirely
6    * independent of the user interface. Thus, potentially implementing this interface. However, the most WComponents deal
7    * with concrete data structures (eg, <code>WTextField</code> is backed by a <code>java.util.String</code>) so in these
8    * cases, implementing this interface is a bit redundant. Where it becomes a benefit is for components that can deal
9    * with an unknown data structure (eg. a reusable panel).
10   * </p><p>
11   * In terms of the MVC pattern, the WComponent represents the View and the data structure/bean represents the Model.
12   * </p><p>
13   * Any <code>WComponent</code> implementing this interface and making use of a Velocity template for layout will have
14   * access to the stored data in the template via the variable <em>$bean</em>.
15   * </p><p>
16   * NOTE: The object returned by {@link #getData()} may not necessarily be the same instance as the one supplied in the
17   * {@link #setData(Object)} call. This is dependant on the implementation.
18   * </p>
19   *
20   * @author Adam Millard
21   */
22  public interface DataBound {
23  
24  	/**
25  	 * Sets the data that this component displays/edits.
26  	 *
27  	 * @param data the data to set.
28  	 */
29  	void setData(Object data);
30  
31  	/**
32  	 * Retrieves the data for the given context.
33  	 *
34  	 * @return the data for the given context.
35  	 */
36  	Object getData();
37  }
